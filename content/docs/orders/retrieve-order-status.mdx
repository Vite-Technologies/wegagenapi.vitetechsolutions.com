## Endpoint

`GET https://pranaconnect.wegagenbanksc.com.et/api/orders/:orderId/status`

Retrieve the real-time processing status of an individual order using the merchant's unique `orderId`. This endpoint is useful when your platform wants to poll transaction status directly or recover after a missed webhook notification.

<Callout title="Recommended usage" type="info">
  Treat this endpoint as a recovery and verification mechanism alongside webhook
  processing, not as a replacement for asynchronous notifications.
</Callout>

- Use this endpoint when your platform needs an explicit status check
- The primary success states are `pending`, `paid`, and `fail`
- Log `requestId` values for failed responses

## Path Parameters

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `orderId` | `string` | Yes | Unique order reference generated by your platform, for example `ORD123456` |

## Implementation Example

```bash
curl --request GET \
  --url https://pranaconnect.wegagenbanksc.com.et/api/orders/ORD123456/status \
  --header 'x-secret-key: pm_test_59ef6a0de47e5e230b56c97f9093b145352e658d86ab19c3d9985f1fc5efe5aa123'
```

## How To Interpret The Result

- Use `pending` when the user has not yet completed authorization
- Use `paid` as the confirmed financial success state
- Use `fail` for explicit customer cancellation, timeout, or unsuccessful execution

## Error Payload

### `404 Not Found`

<Callout title="Not found handling" type="warning">
  Use the returned `requestId` and `message` fields to troubleshoot missing
  order references before escalating to support.
</Callout>

If the requested `orderId` does not match an order in the services cluster, the API returns a not found payload.

```json
{
  "statusCode": 404,
  "timestamp": "2026-06-27T23:30:12.004Z",
  "path": "/api/orders/ORD_UNKNOWN/status",
  "requestId": "req_cls_e7f8a9b0c1d2",
  "code": "INTERNAL_ERROR",
  "message": "order not found"
}
```

## Notes

- Use this endpoint as a recovery and verification path alongside webhook delivery
- Expect `pending`, `paid`, or `fail` as the primary transaction status states
- Persist `requestId` values for support and audit troubleshooting
